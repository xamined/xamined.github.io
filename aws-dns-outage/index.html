What Happened

Two automated jobs ran at the same time inside AWS's internal DNS system. A cleanup script removed stale records. A separate automation job re-registered them using cached (now outdated) data. The automation job finished last, so the stale records won. DynamoDB couldn't find its own nodes.
10:45 UTC
Cleanup script begins removing old internal DNS entries in US-EAST-1.
~10:46 UTC
Automation job starts re-registering entries using cached data. Both jobs are now writing to the same DNS records.
~10:50 UTC
Automation job finishes after the cleanup. Stale entries overwrite the correct ones. DynamoDB nodes become unreachable.
11:00 UTC
DynamoDB API errors spike. Lambda, S3, CloudWatch, and EventBridge all depend on DynamoDB. They start failing too.
11:15 UTC
AWS status page (also hosted on AWS) goes stale. External monitoring is the only source of information.
~02:00 UTC+1
Engineers identify the DNS race condition and begin manually correcting records.
~04:00 UTC+1
Services begin recovering. Full recovery takes several more hours as caches expire.
